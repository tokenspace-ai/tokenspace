title: Capabilities
description: Author capability actions with narrow scope, explicit inputs, and safe autonomy.
---

Capabilities are the core building blocks of a tokenspace. They define the actions an AI can call against external systems.

Each capability lives under `src/capabilities/<name>/` and should include:

```text
src/
  capabilities/
    github/
      CAPABILITY.md
      capability.ts
```

## Start with the domain contract

Before writing code, define what the capability is allowed to touch.

- Which orgs, tenants, repos, environments, or projects are in scope?
- What identifier formats are allowed?
- Which targets must always be rejected?

If scope is ambiguous, choose the narrower boundary. The capability-authoring workflow in this repo explicitly prefers strict domain gating over convenience.

## Design small action surfaces

Prefer small, composable actions over one broad "do anything" entrypoint.

- Read actions should usually stay autonomous.
- Reversible writes may stay autonomous if the workspace policy allows it.
- Destructive, costly, or high-impact writes should require approval.
- Out-of-domain operations should be rejected, not approved.

## Implement actions with runtime validation

Use `action(...)` from `@tokenspace/sdk` so inputs are parsed and both inputs and outputs stay JSON-serializable.

```typescript
import { action, getCredential, requireApproval, TokenspaceError } from "@tokenspace/sdk";
import z from "zod";
import { githubToken } from "../../credentials";

const createIssueInput = z.object({
  owner: z.string(),
  repo: z.string(),
  title: z.string().min(1),
  body: z.string().optional(),
});

function assertAllowedRepo(args: z.infer<typeof createIssueInput>) {
  const allowedRepos = new Set(["platform/api", "platform/web"]);
  const fullName = `${args.owner}/${args.repo}`;
  if (!allowedRepos.has(fullName)) {
    throw new TokenspaceError(`Out-of-domain repository: ${fullName}`);
  }
}

export const createIssue = action(createIssueInput, async (args) => {
  assertAllowedRepo(args);
  const token = await getCredential(githubToken);

  await requireApproval({
    action: "github:createIssue",
    data: { owner: args.owner, repo: args.repo },
    description: `Create issue in ${args.owner}/${args.repo}`,
  });

  const response = await fetch(`https://api.github.com/repos/${args.owner}/${args.repo}/issues`, {
    method: "POST",
    headers: {
      Authorization: `Bearer ${token}`,
      "Content-Type": "application/json",
    },
    body: JSON.stringify({
      title: args.title,
      body: args.body,
    }),
  });

  return (await response.json()) as { number: number; html_url: string };
});
```

## Capability authoring rules that matter

- Validate untrusted user input before external calls.
- Re-check scope before mutating operations, not just at action entry.
- Normalize names and IDs before comparisons when case or aliases matter.
- Keep action input and output plain JSON values.
- Use least-privilege credentials for each operation.

## Write `CAPABILITY.md` for the agent

`CAPABILITY.md` is not decorative. It is part of the agent's operating context and should document:

- when to use the capability
- scope and guardrails
- available read and write actions
- concrete examples
- at least one rejected out-of-domain example

A useful shape is:

```markdown
---
name: GitHub
description: GitHub actions for the platform-owned repositories only.
---

## When to Use This Capability

## Scope and Guardrails

## Available Operations

## Guidelines

## Data and Workflow Context

## Examples
```

## Practical checklist

1. Define the allowed domain before implementation.
2. Keep actions narrow and typed.
3. Reject unsupported targets explicitly.
4. Add approval gates only where risk justifies them.
5. Keep `CAPABILITY.md` aligned with the actual exported actions.
