---
title: Credentials
description: Define workspace credentials in code and resolve them at runtime with explicit scope.
---

TokenSpace credential requirements are defined in `src/credentials.ts`.

The current SDK supports three credential kinds:

| Kind | Use for |
| --- | --- |
| `secret` | Stored secret values such as API keys or client secrets |
| `env` | Executor environment variables |
| `oauth` | OAuth tokens acquired for workspace, session, or user scope |

## Define credentials in code

```typescript
import { credentials } from "@tokenspace/sdk";

export const githubClientSecret = credentials.secret({
  id: "github-client-secret",
  label: "GitHub client secret",
  scope: "workspace",
});

export const githubAppToken = credentials.oauth({
  id: "github-app-token",
  label: "GitHub OAuth token",
  scope: "user",
  config: {
    grantType: "authorization_code",
    clientId: "github-oauth-client-id",
    clientSecret: credentials.ref(githubClientSecret),
    authorizeUrl: "https://github.com/login/oauth/authorize",
    tokenUrl: "https://github.com/login/oauth/access_token",
    scopes: ["repo", "read:user"],
  },
});

export const datadogApiKey = credentials.env({
  id: "datadog-api-key",
  label: "Datadog API key",
  variableName: "DATADOG_API_KEY",
});
```

## Credential scopes

| Scope | Meaning |
| --- | --- |
| `workspace` | Shared across users of the tokenspace |
| `session` | Available only for one session |
| `user` | Bound to one user's identity and permissions |

Choose the narrowest scope that matches the external system contract.

## Consume credentials in a capability

```typescript
import { action, getCredential } from "@tokenspace/sdk";
import z from "zod";
import { githubAppToken } from "../../credentials";

export const listPullRequests = action(
  z.object({
    owner: z.string(),
    repo: z.string(),
  }),
  async (args) => {
    const oauth = await getCredential(githubAppToken);
    const token = oauth.accessToken;
    return { owner: args.owner, repo: args.repo, tokenPreview: token.slice(0, 4) };
  },
);
```

## Operational behavior

- Workspace-scoped credentials are configured by admins in the web app or with the [CLI](/docs/tokenspace-cli#manage-credentials) (`tokenspace credentials set`).
- Session and user credentials are usually resolved when a run asks for them.
- Missing required credentials fail explicitly instead of degrading silently.
- `env` credentials are only practical with self-hosted executors, because the value must exist in the executor process environment.
- Use `tokenspace credentials list` to see all declared credentials, their scopes, and configuration status.

## Authoring guidance

- Keep credential IDs unique within the workspace.
- Prefer `credentials.ref(...)` when an OAuth definition needs a stored client secret.
- Use `optional: true` only when the capability can degrade safely.
- Do not broaden capability scope just because the credential itself has broad access.
