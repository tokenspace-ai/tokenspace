The TokenSpace CLI allows you to create and build tokenspaces locally, link them to remote tokenspaces, sync source files, compile locally, and push revisions.

## Installation

```bash
bun install tokenspace -g
```

or use adhoc

```bash
bunx tokenspace <command>
```

## Authentication

The CLI uses WorkOS device authorization.

```bash
tokenspace login
tokenspace login --url=http://localhost:31337
tokenspace whoami
tokenspace logout
```

Auth state is stored locally in `~/.tokenspace/auth.json`. The CLI remembers the selected web app URL, discovers the matching WorkOS client ID and Convex URL automatically, and reuses that target for future commands and generated links.

Choose a default workspace for commands run outside a linked workspace directory:

```bash
tokenspace use my-workspace
tokenspace use
```

`tokenspace use` stores the default workspace slug in `~/.tokenspace/auth.json`. Commands that support workspace selection, such as `tokenspace chat ...`, resolve the workspace in this order: `--workspace`, current linked workspace, then the default set by `tokenspace use`.

## Standard workflow

For a hosted tokenspace workflow:

```bash
tokenspace init
cd ./your-workspace
tokenspace login
tokenspace link --create --name "Your Workspace" --slug your-workspace
tokenspace push
```

After the first link, you can keep working from that directory with `tokenspace pull`, `tokenspace push`, and `tokenspace compile`.

## Link a workspace locally

Link the current directory to an existing tokenspace or create a new one:

```bash
tokenspace link
tokenspace link my-workspace
tokenspace link --create --name "Platform Ops" --slug platform-ops
```

The CLI stores the link in `.tokenspace/link.json` and adds `.tokenspace/` to `.gitignore` if needed.

Workspace slugs cannot contain `:` or `@` because those delimiters are reserved for branch and revision URLs in the app.

## Sync a workspace locally

Pull downloads the current branch state, including your working changes:

```bash
tokenspace pull
tokenspace pull --dry-run
```

Push uploads your local files into the linked tokenspace's default-branch working directory, builds local artifacts, and pushes a revision:

```bash
tokenspace push
tokenspace push --dry-run
tokenspace push --open
```

Successful pushes print a revision playground URL in the form `/workspace/<slug>@<revisionId>/playground`.

The CLI skips hidden files and local-only directories such as `.tokenspace/`, `.git/`, `node_modules/`, and `build/`.

## Compile locally

Compile the current local tokenspace into revision artifacts:

```bash
tokenspace compile
tokenspace compile --out-dir build/tokenspace
```

## Chat from the CLI

Launch the interactive terminal chat UI:

```bash
tokenspace chat
tokenspace chat --workspace my-workspace
tokenspace chat --model openai:gpt-5.4 --open
```

Running `tokenspace chat` opens a persistent terminal chat with a single-line composer. The first submitted prompt lazily creates a chat, streams updates live in the terminal, and keeps the same thread open so you can continue once the assistant turn completes.

The interactive UI follows the same workspace resolution order as the other chat commands: `--workspace`, current linked workspace, then the default set by `tokenspace use`.

Human-readable chat output is compact and follows the same step order as the web UI. The CLI renders:

- user messages in blue
- assistant messages in your terminal's normal foreground color
- reasoning in dim gray
- tool calls in dim gray with a leading `•` and simplified labels like `Read /sandbox/README.md` or `Running code`

Example:

```text
Inspect the repo
Looking through the files
• Read /sandbox/README.md
• Check the compiled artifact
Found the relevant section.
```

For scripting and one-shot terminal usage, the non-interactive subcommands remain available.

Start a new chat:

```bash
tokenspace chat start "Summarize the credential setup"
tokenspace chat start --stdin --follow
tokenspace chat start --workspace my-workspace "Review the latest logs"
```

List chats:

```bash
tokenspace chat list
tokenspace chat list --workspace my-workspace
tokenspace chat list --all --json
```

Inspect a chat or follow it in the terminal:

```bash
tokenspace chat get <chat-id>
tokenspace chat get <chat-id> --follow
tokenspace chat get <chat-id> --json
tokenspace chat get <chat-id> --follow --ndjson
```

Send another user message to an existing chat:

```bash
tokenspace chat send <chat-id> "Continue from the last result"
tokenspace chat send <chat-id> --stdin --follow
```

These commands require an existing compiled revision for the selected workspace. If the selected tokenspace has not been pushed yet, run `tokenspace push` first.

## Manage credentials

List the credential requirements declared by the current compiled revision:

```bash
tokenspace credentials list
```

This shows workspace-scoped and runtime-scoped credentials, their kinds (secret, env, oauth), and whether each workspace credential has been configured. Admin access is required to see configuration status.

Set a workspace-scoped secret credential:

```bash
tokenspace credentials set <credential-id>
```

The CLI prompts for the secret value interactively. For non-interactive use (CI, scripts), pipe the value via `--stdin`:

```bash
echo "sk-abc123" | tokenspace credentials set my-api-key --stdin
```

Only workspace-scoped secret credentials can be set through the CLI. OAuth credentials must be connected through the web app, and env credentials are provided by the executor environment.

## When to use the CLI vs the UI

Use the CLI when:

- you want the workspace files in a local editor
- you want a local-first workflow that still pushes revisions to the app
- you want to compile the workspace locally from the CLI
- you want to list or set workspace credentials from the terminal or a script

Use the UI when:

- you want browser-based editing and review
- you are managing OAuth credentials or approvals
- you want to run and inspect chats directly in the product
